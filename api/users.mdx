The Users API provides endpoints for creating, updating, and retrieving user information. User operations follow a maker-checker workflow for enhanced security and compliance.

## Create User

<CodeGroup>
```http Request
POST /api/users
```
</CodeGroup>

Creates a new user in the system. This operation follows the maker-checker pattern and creates a request that must be approved.

### Request Body

<ParamField body="username" type="string" required>
  Unique username for login
</ParamField>

<ParamField body="password" type="string" required>
  User password (will be encrypted)
</ParamField>

<ParamField body="firstName" type="string" required>
  User's first name
</ParamField>

<ParamField body="lastName" type="string" required>
  User's last name
</ParamField>

<ParamField body="roleId" type="long" required>
  ID of the role to assign to the user
</ParamField>

<ParamField body="branchId" type="integer" required>
  ID of the branch the user belongs to
</ParamField>

<ParamField body="email" type="string">
  User's email address
</ParamField>

<ParamField body="phoneNumber" type="string">
  User's phone number
</ParamField>

<ParamField body="idNumber" type="string">
  User's identification number
</ParamField>

<ParamField body="address" type="string">
  User's physical address
</ParamField>

<ParamField body="position" type="string">
  User's job position/title
</ParamField>

<ParamField body="expireDate" type="string">
  ISO 8601 date when the user account expires
</ParamField>

<ParamField body="statusType" type="string" default="ACTIVE">
  User status (ACTIVE, INACTIVE, etc.)
</ParamField>

<CodeGroup>
```json Request Example
{
  "username": "jsmith",
  "password": "SecurePass123!",
  "firstName": "John",
  "lastName": "Smith",
  "roleId": 2,
  "branchId": 1,
  "email": "john.smith@example.com",
  "phoneNumber": "+1-555-0123",
  "idNumber": "ID123456",
  "address": "123 Main St, City, Country",
  "position": "Loan Officer",
  "expireDate": "2027-12-31",
  "statusType": "ACTIVE"
}
```
</CodeGroup>

### Response

Returns a request ID that can be used to track the approval process.

<ResponseField name="id" type="long">
  Unique identifier for the maker-checker request
</ResponseField>

<CodeGroup>
```json Response Example
{
  "id": 42
}
```

```json Error Response (400)
{
  "message": "Validation failed",
  "errors": [
    {
      "field": "username",
      "message": "Username already exists"
    }
  ],
  "status": 400
}
```
</CodeGroup>

## Update User

<CodeGroup>
```http Request
PUT /api/users/{id}
```
</CodeGroup>

Updates an existing user. This operation follows the maker-checker pattern and creates a request that must be approved.

### Path Parameters

<ParamField path="id" type="long" required>
  The unique identifier of the user to update
</ParamField>

### Request Body

Accepts the same fields as the Create User endpoint (excluding password, which should be updated via the update-password endpoint).

<CodeGroup>
```json Request Example
{
  "username": "jsmith",
  "firstName": "John",
  "lastName": "Smith",
  "roleId": 3,
  "branchId": 1,
  "email": "john.smith@newdomain.com",
  "phoneNumber": "+1-555-0123",
  "position": "Senior Loan Officer",
  "statusType": "ACTIVE"
}
```
</CodeGroup>

### Response

Returns a request ID for the maker-checker approval process.

<ResponseField name="id" type="long">
  Unique identifier for the maker-checker request
</ResponseField>

<CodeGroup>
```json Response Example
{
  "id": 43
}
```

```json Error Response (404)
{
  "message": "User not found (ID=999).",
  "status": 404
}
```
</CodeGroup>

## Update Password

<CodeGroup>
```http Request
PUT /api/users/update-password
```
</CodeGroup>

Updates a user's password.

### Request Body

<ParamField body="userId" type="long" required>
  ID of the user whose password is being updated
</ParamField>

<ParamField body="oldPassword" type="string">
  Current password (required when users update their own password)
</ParamField>

<ParamField body="newPassword" type="string" required>
  New password
</ParamField>

<CodeGroup>

```json Request Example
{
  "userId": 5,
  "oldPassword": "OldPass123!",
  "newPassword": "NewSecurePass456!"
}
```
</CodeGroup>

### Response

Returns the updated user object.

<CodeGroup>
```json Response Example
{
  "id": 5,
  "username": "jsmith",
  "firstName": "John",
  "lastName": "Smith",
  "statusType": "ACTIVE"
}
```

```json Error Response (400)
{
  "message": "Validation failed",
  "errors": [
    {
      "field": "oldPassword",
      "message": "Current password is incorrect"
    }
  ],
  "status": 400
}
```
</CodeGroup>

## Permissions

User creation and update operations require the `MAKER_FOR_USER` permission with the `MAKER_CHECKER` module type. This implements a two-step approval process where one user creates/updates and another approves the changes.

## Notes

- User ID 1 (system admin) cannot be updated or deleted
- Password updates follow separate validation rules and do not require maker-checker approval
- System users (internal service accounts) are excluded from lookup endpoints
