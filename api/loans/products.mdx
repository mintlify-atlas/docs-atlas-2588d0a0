## Overview

Loan Products define the templates and rules for creating loans in OpenCBS Cloud. Each product specifies:

- Interest rates and calculation methods
- Maturity ranges and grace periods
- Entry fees and penalties
- Accounting rules for financial transactions
- Eligibility criteria and availability
- Top-up configurations

All loan product creation and modification operations use the maker-checker approval workflow for compliance and control.

```
POST /api/loan-products
```

Create a new loan product. This operation creates a maker-checker request that must be approved before the product becomes active.

**Permission Required**: `MAKER_FOR_LOAN_PRODUCT`

#### Request Body

<ParamField body="name" type="string" required>
  Product name (must be unique)
</ParamField>

<ParamField body="code" type="string" required>
  Product code for reference (must be unique)
</ParamField>

<ParamField body="currencyId" type="integer" required>
  Currency ID for the product
</ParamField>

<ParamField body="amountMin" type="decimal" required>
  Minimum loan amount
</ParamField>

<ParamField body="amountMax" type="decimal" required>
  Maximum loan amount (must be >= amountMin)
</ParamField>

<ParamField body="interestRateMin" type="decimal" required>
  Minimum annual interest rate (percentage)
</ParamField>

<ParamField body="interestRateMax" type="decimal" required>
  Maximum annual interest rate (percentage, must be >= interestRateMin)
</ParamField>

<ParamField body="maturityMin" type="integer" required>
  Minimum loan maturity in schedule periods
</ParamField>

<ParamField body="maturityMax" type="integer" required>
  Maximum loan maturity in schedule periods (must be >= maturityMin)
</ParamField>

<ParamField body="gracePeriodMin" type="integer" required>
  Minimum grace period in schedule periods
</ParamField>

<ParamField body="gracePeriodMax" type="integer" required>
  Maximum grace period in schedule periods (must be >= gracePeriodMin)
</ParamField>

<ParamField body="scheduleBasedType" type="string" required>
  Schedule calculation basis: `DAYS`, `WEEKS`, `MONTHS`, or `YEARS`
</ParamField>

<ParamField body="availability" type="array" required>
  Array of profile types that can access this product: `PERSON`, `COMPANY`, `GROUP`
</ParamField>

<ParamField body="hasPayees" type="boolean" required>
  Whether the product supports multiple payees
</ParamField>

<ParamField body="topUpAllow" type="boolean">
  Whether top-up loans are allowed
</ParamField>

<ParamField body="topUpMaxLimit" type="decimal">
  Maximum top-up amount (required if topUpAllow is true)
</ParamField>

<ParamField body="fees" type="array">
  Array of entry fee configurations
</ParamField>

<ParamField body="penalties" type="array">
  Array of penalty configurations
</ParamField>

<ParamField body="accounts" type="array">
  Array of accounting rule mappings
</ParamField>

#### Response

Returns a `RequestDto` containing the maker-checker request ID.

<ResponseField name="id" type="integer">
  Maker-checker request ID. This request must be approved by an authorized checker before the product is created.
</ResponseField>

<CodeGroup>
```json 200 Response
{
  "id": 42
}
```
</CodeGroup>

## Best Practices

### Product Configuration

1. **Logical Ranges**: Ensure min/max ranges are logical and don't overlap inappropriately
   - `amountMax >= amountMin`
   - `interestRateMax >= interestRateMin`
   - `maturityMax >= maturityMin`

2. **Currency Consistency**: All monetary amounts must be in the product's configured currency

3. **Accounting Rules**: Configure all required account rules before activating a product:
   - `PRINCIPAL`
   - `INTEREST`
   - `INTEREST_ACCRUAL`
   - `PENALTY`

4. **Testing**: Test product configuration with sample loan applications before going live

### Maker-Checker Workflow

1. **Separation of Duties**: Ensure makers and checkers are different users
2. **Request Review**: Checkers should carefully review all product changes before approval
3. **Documentation**: Include comments in maker-checker requests explaining the reason for changes

### Versioning

1. **Audit Trail**: Use the history endpoints to maintain compliance and audit trails
2. **Impact Analysis**: Before updating a product, check how many active loans/applications use it
3. **Communication**: Notify loan officers of product changes that affect their portfolios
